# РЕЛИЗ 1: АВТОПЕРЕВОЗКИ КОНТЕЙНЕРОВ

**White Label COUBE для Keden Trans Service**

**Дата:** 10.02.2026

---

## ЦЕЛЬ РЕЛИЗА

Полностью автоматизировать автотранспортный цикл контейнерных перевозок:

**Терминал → Автомобиль → Склад клиента → Терминал (сдача порожнего)**

Создать "ядро TMS" COUBE White Label, обеспечивающее 100% цифровой учёт контейнеров, пробега, топлива, инцидентов и KPI.

---

## ТИПЫ ПЕРЕВОЗОК РЕЛИЗА 1

1. **Гружёный контейнер: терминал → склад клиента** (включает 2 этапа: доставка гружёного + возврат порожнего)
2. **Гружёный контейнер: склад → терминал** (забор контейнера и доставка для ЖД-отправки)
3. **Порожние контейнеры: терминал ↔ терминал** (городские потоковые перевозки для формирования поездов)
4. **Междугородние перевозки фурами (тенты)** (FTL — уже поддерживается COUBE)

## 2. МОДУЛИ РЕЛИЗА 1: ДЕТАЛЬНОЕ ОПИСАНИЕ

### Модуль 1: Заявки на автодоставку контейнеров

**Готовность: 70%**

**Функционал:**
- Выбор типа перевозки:
  - Гружёный контейнер с терминала → склад
  - Контейнер со склада → терминал
  - Порожний контейнер терминал → терминал
- Автоматическое формирование состава заявки:
  - номер контейнера, тип, вес, владелец
  - терминал хранения
  - склад назначения
  - необходимость комиссионной выдачи
- Проверка статуса разрешений Clearance
- Запрет назначения машины, если нет разрешений

**Что нужно сделать:**
Добавить типы контейнерных перевозок (гружёный→склад, склад→терминал, порожний→терминал). Привязка контейнера к заявке. Проверка Clearance перед назначением машины.

### Модуль 2: Container ID — строгая идентификация контейнера

**Готовность: 0% — разработка с нуля**

**Проблема (из SFM):**
При автоперевозке и терминальной обработке контейнеров существует риск неправильной идентификации. Контейнеры часто проверяются визуально и не полностью — по последним цифрам номера (которые могут совпасть у разных контейнеров). Возможна ситуация, когда под погрузку попадает контейнер, не соответствующий заявке.

**Цель:**
Обеспечить однозначную и обязательную идентификацию контейнера по полному номеру на всех этапах. Контроль: доля операций с подтверждённой сверкой — **100%**.

**Функционал:**
1. **Цифровой профиль контейнера** — номер (полный), владелец, тип (20ft/40ft/HC/рефрижератор), статус (гружёный/порожний/в пути/готов к вывозу/ожидание разрешений), история операций
2. **Обязательная сверка полного номера** при: погрузке, выезде с терминала, сдаче порожняка
3. **Сканирование QR / RFID / камера** в мобильном приложении водителя
4. **Блокировка операций** при несовпадении номера
5. **Единый источник правды** для водителя, терминала, диспетчера
6. **Протокол всех операций** — время, место, ответственный

### Модуль 3: Dispatching — назначение транспорта

**Готовность: 60%**

**Функционал:**
- Выбор собственного ТС или подрядчика
- Фильтрация: грузоподъёмность, тип тягача, прицеп
- Назначение водителя с проверкой допуска (медосмотр)
- Автоматическая проверка графика работы водителя

**Что нужно сделать:**
Фильтрация по грузоподъёмности/типу тягача/прицепу. Проверка мед. допуска при назначении. Проверка графика работы водителя. Выбор собственный ТС vs подрядчик.

### Модуль 4: Driver Management — медосмотр и допуск водителя

**Готовность: 15% — разработка с нуля**

**Проблема (из SFM):**
Существует риск неконтролируемого допуска водителей к рейсам. Водитель может быть уставшим, находиться в неадекватном состоянии. При отсутствии обязательного медицинского контроля существует риск выхода водителя в рейс в состоянии, не позволяющем безопасно управлять ТС.

**Цель:**
Допуск к рейсам только медицински допущенных водителей. Контроль: доля рейсов в нормативном времени — **не менее 90%**, нарушений скорости на 1000 км — **не более 0**.

**Функционал:**
1. **Мед. допуск** — подтверждение медработником (мобильный/веб)
2. **Блокировка** — назначение на рейс невозможно без мед-OK
3. **Журнал допусков** — история всех допусков
4. **Учёт перевозок по водителю** — не только по машине
5. **Гибкость привязки** — водитель может быть назначен на любой доступный автомобиль
6. **Фиксация нарушений** — скорость, отклонения от маршрута, простои
7. **KPI водителя** — на основе данных системы
8. **История работы** — рейсы, простои, инциденты

---

### Модуль 5: Slot & Queue — электронная очередь на терминал

**Готовность: 0% — разработка с нуля**

**Проблема (из SFM):**
Машины могут приезжать одновременно, заезжать без очереди, опаздывать к своему времени. Часть машин простаивает, возникают конфликты, заторы на территории терминала, а перегрузочная техника используется неэффективно.

**Цель:**
Упорядоченный и предсказуемый заезд машин на терминал. Контроль: среднее время ожидания под погрузку — **не более 15 минут**, доля машин, перераспределённых без простоя — **не менее 10%**.

**Функционал:**
1. **Назначение слота** — конкретное время заезда и погрузки для каждой машины
2. **Электронная очередь** — заезд только в порядке очерёдности
3. **Опоздание не останавливает очередь** — очередь продолжает движение
4. **Приоритет собственного автопарка** — автоматическая замена при опоздании
5. **Правила для сторонних машин** — встают в очередь по факту прибытия
6. **Готовность терминала** — информация заранее передаётся крановщику
7. **Прозрачная очередь** — водители и диспетчеры видят свою очередь в реальном времени
8. **Фиксация опозданий** — все срывы слотов и замены фиксируются
9. **Отображение в мобилке водителя** — текущий слот и очередь

---

### Модуль 6: ETA / Loading & Handling — погрузка контейнера

**Готовность: 35%**

**Проблема (из SFM):**
Крановщик и оператор заранее не знают, какая машина к ним приедет, за каким контейнером и когда. Время прибытия не рассчитывается автоматически. Диспетчер вынужден постоянно звонить водителю. Погрузка происходит «по факту», а не по плану.

**Цель:**
Синхронная работа водителя, диспетчера и терминала. Контроль: доля погрузок в плановом окне — **не менее 70%**.

**Функционал:**
1. **Предварительное назначение** — машина ↔ контейнер заранее в системе
2. **Автоматический расчёт ETA** до терминала по GPS
3. **Отображение приближения** для крановщика и терминальных служб
4. **Подготовка контейнера заранее** по ETA
5. **Замена машины/контейнера** через систему при пробое слота
6. **Фиксация погрузки** — начало, окончание, фото контейнера
7. **Минимизация звонков** — диспетчеру не нужно уточнять местоположение

---

### Модуль 7: Route & Tracking — контроль рейса в реальном времени

**Готовность: 75%**

**Проблема (из SFM):**
Диспетчер и клиент не всегда понимают, где находится машина, движется ли она по маршруту или стоит без причины. Водитель может умышленно затягивать рейс. Вся информация передаётся устно по телефону.

**Цель:**
Прозрачный контроль рейса без звонков. Контроль: доля рейсов без телефонных уточнений — **не менее 90%**, ручных контактов на рейс — **не более 3**.

**Функционал:**
1. **Онлайн GPS-трек** — местоположение в реальном времени на карте
2. **Этапы контейнерного рейса:**
   - в пути → прибыл → выгрузка → выехал → сдача порожнего → завершено
3. **Прогноз ETA** с учётом пробок и отклонений
4. **Видимость для клиента** — движение + ETA + подготовка склада
5. **Контроль отклонений** — простои, отклонения от маршрута, превышение скорости
6. **Автоклассификация простоев** — рабочие задержки vs необоснованные
7. **Основа для мотивации** — данные о фактическом движении для оценки работы водителя

---

### Модуль 8: Warehouse Delivery — работа на складе клиента

**Готовность: 40%**

**Проблема (из SFM):**
Склад клиента часто не готов к приёму машины: бригада не собрана, ворота заняты, нет информации о времени прибытия. Машина простаивает, возникают споры о причинах задержки. При отсутствии фиксации задержка может быть ошибочно отнесена на водителя.

**Цель:**
Прозрачный процесс приёма и выгрузки с фиксацией всех этапов. Контроль: доля рейсов с зафиксированным временем выгрузки — **100%**.

**Функционал:**
1. **Фиксация прибытия** — автоматически (GPS) или по подтверждению водителя
2. **Проверка пломб** — фото/видео контейнера
3. **Статус «комиссионная выдача»** — при повреждении контейнера или подозрении на повреждение груза
4. **Фиксация выгрузки** — начало и окончание
5. **Классификация задержек:**
   - вина склада
   - вина водителя
   - объективные риски
6. **Подтверждение сдачи груза** в системе

### Модуль 9: Empty Container Return — сдача порожних контейнеров

**Готовность: 30%**

**Проблема (из SFM):**
Адрес сдачи может передаваться водителю неофициально (в переписке). Подтверждение сдачи оформляется на бумаге и может быть утеряно. При отсутствии фиксации перевозка формально считается незавершённой, возникают споры.

**Цель:**
Прозрачный процесс сдачи порожнего контейнера. Контроль: доля возвратов, подтверждённых в системе — **100%**, подтверждённых в день сдачи — **не менее 90%**.

**Функционал:**
1. **Адрес сдачи в системе** — терминал/депо, не устно
2. **Навигация водителя** к точке сдачи
3. **Фото-подтверждение сдачи** — фото штампа или контейнера на территории
4. **Журнал сдачи** — привязка к рейсу, машине, водителю
5. **Завершение рейса** — перевозка закрыта **только после сдачи порожняка**

---

### Модуль 10: Fuel & Vehicle Integrity — контроль топлива и пробега

**Готовность: 15% — разработка с нуля**

**Проблема (из SFM):**
Водитель может завышать пробег (вмешательство в одометр), получать больше топлива, чем требуется, списывать топливо без фактического движения. Ситуации сложно выявлять постфактум и они приводят к прямым финансовым потерям.

**Цель:**
Прозрачный контроль пробега и топлива. Контроль: совпадение GPS-пробега с расчётным — **не менее 99%**, отклонение расхода от норматива — **не более 10%**.

**Функционал:**
1. **Пробег только по GPS** — одометр не учитывается, ручной ввод исключён
2. **Норма топлива автоматически** — по маршруту, расстоянию и типу ТС
3. **Привязка заправок к рейсу** — время, место, объём, в пределах лимита
4. **Контроль сливов** — датчики уровня топлива при наличии
5. **Датчики CAN** — при интеграции с IoT/ГЛОНАСС
6. **Фиксация тех. событий** — вскрытие баков, отключение датчиков
7. **Аналитика аномалий** — нетипичные маршруты, перерасход, подозрительные остановки

---

### Модуль 11: Incident & Exception — управление инцидентами

**Готовность: 25%**

**Проблема (из SFM):**
Могут возникать повреждённые контейнеры, комиссионная выдача, поломки, пробки, аварии, вскрытие контейнера. При отсутствии фиксации причины задержек неочевидны. Водитель может быть ошибочно признан виновным.

**Цель:**
Обязательная фиксация всех инцидентов. Контроль: доля инцидентов, зафиксированных в день возникновения — **100%**, рейсов с классифицированной причиной простоя — **100%**.

**Функционал:**
1. **Фиксация в момент возникновения** — водитель, диспетчер или другой участник
2. **Категории инцидентов:**
   - повреждённый контейнер
   - комиссионная выдача
   - поломка автомобиля
   - ДТП
   - пробка
   - отсутствие топлива
   - задержка склада
   - вскрытие контейнера
3. **Привязка к рейсу** — рейс, машина, водитель, контейнер
4. **Фиксация времени и длительности**
5. **Фото/видео подтверждение**
6. **Разделение ответственности** — инциденты, не зависящие от водителя, не влияют на его оценку

---

### Модуль 12: Аналитика и KPI (авто)

**Готовность: 50%**

**Проблема (из SFM):**
Данные о рейсах, простоях, нарушениях, доходах и расходах разрознены. Решения принимаются на основе ощущений. Система оплаты труда субъективна и вызывает недоверие.

**Цель:**
Прозрачный учёт всех перевозок. Контроль: доля рейсов в автоматическом расчёте — **100%**, зарплатных расчётов автоматически — **100%**.

**Отчёты:**
- Время рейса (план/факт)
- Простои с классификацией причин
- KPI водителей и машин
- Топливо (норма/факт)
- Инциденты
- Статистика выполнения слотов
- Средняя скорость
- Маржинальность рейса
- Показатели на 1 контейнер / 1 км

**Мотивация:**
- Автоматический расчёт ЗП, премий, бонусов по данным системы
- Прозрачность по ролям: водитель / диспетчер / менеджер / клиент

## 3. WHITE LABEL ИНФРАСТРУКТУРА (РЕЛИЗ 1)

### Бренд и интерфейс
- SCSS-тема Keden (цвета, логотип, favicon)
- Домен: platform.keden.kz
- Кастомные тексты и лейблы

### Изоляция данных
- Отдельный Keycloak realm `keden-trans-service`
- Отдельная PostgreSQL-база (полная изоляция)
- Отдельный S3-бакет для файлов Keden
- Отдельные Docker-контейнеры (бэк + фронт + мобилка)
- CI/CD для отдельной сборки

### Роли

| Роль Keden | Роль COUBE | Действие |
|------------|-----------|----------|
| Администратор Keden | SUPER_ADMIN + ADMIN | Готово |
| Менеджер Keden | CEO / ADMIN | Готово |
| Диспетчер Keden | LOGISTICIAN → **DISPATCHER** | Расширить / создать |
| Оператор терминала | — | **Новая роль: TERMINAL_OPERATOR** |
| Оператор склада клиента | — | **Новая роль: WAREHOUSE_OPERATOR** |
| Водитель | DRIVER | Готово |

## 5. ОЦЕНКА СРОКОВ

### Команда: 12 человек
4 бэкенд, 3 фронтенд, 2 мобильных, 2 QA, 1 DevOps

### Предпосылки
- Зрелая кодовая база: GPS, карты, уведомления, отчёты — переиспользуются (экономия ~40%)
- Максимальная параллелизация: все блоки — разными парами/тройками
- ИИ-инструменты для ускорения (генерация CRUD, миграции, компоненты)
- Непрерывное QA с недели 2

### Блоки разработки

| Блок | Что входит | Команда | Длительность |
|------|-----------|---------|-------------|
| **WL-инфра** | Keycloak, Docker, тема Keden, домен, S3, роли | DevOps + 1 бэк + 1 фронт + 1 моб | **1 нед.** |
| **Контейнер + Clearance** | Container ID, сканирование, разрешения, блокировки | 1 бэк + 1 фронт + 1 моб | **3–4 нед.** |
| **Dispatching + Driver Mgmt** | Назначение ТС, мед. допуск, журнал, KPI водителя | 1 бэк + 1 фронт + 0.5 моб | **2–3 нед.** |
| **Slot & Queue + Loading** | Слоты, очередь, ETA, погрузка, фото | 1 бэк + 1 фронт + 0.5 моб | **3–4 нед.** |
| **Route + Warehouse + Порожний** | Этапы рейса, ETA, выгрузка, пломбы, сдача | 1 бэк + 0.5 фронт + 1 моб | **2–3 нед.** |
| **Fuel + Инциденты** | Топливо, GPS-пробег, инциденты, фото | 0.5 бэк + 0.5 фронт + 0.5 моб | **2–3 нед.** |
| **Аналитика (авто)** | Отчёты: план/факт, KPI, топливо, слоты, маржа | 0.5 бэк + 0.5 фронт | **2–3 нед.** |
| **QA** | Тестирование каждого блока по мере готовности | 2 QA | **параллельно** |
