## 2. ЧТО ТРЕБУЕТ SFM vs ЧТО ЕСТЬ

### FLOW: Полный цикл перевозки контейнера SFM

```
ТЕРМИНАЛ (ж/д)
    │
    ├─── Прибытие контейнера → Оформление → Растаможка → Оплата услуг
    │
    ├─── [РАЗРЕШЕНИЯ] Цифровой статус: СВХ + КТЖ + КДТС + ТХК ─── все ОК?
    │
    ├─── [ЗАКАЗ МАШИНЫ] Только после получения всех разрешений
    │
    ├─── [СЛОТ/ОЧЕРЕДЬ] Назначение времени заезда → Электронная очередь
    │
    ├─── [ЗАЕЗД НА ТЕРМИНАЛ] Проверка разрешений → Сверка контейнера
    │
    ├─── [ПОГРУЗКА] ETA машины → Подготовка контейнера → Погрузка → Фиксация
    │
    ├─── [В ПУТИ] GPS-трекинг → ETA для клиента → Контроль отклонений
    │
    ├─── [СКЛАД КЛИЕНТА] Прибытие → Проверка пломб → Выгрузка → Сдача груза
    │
    ├─── [СДАЧА ПОРОЖНЕГО] Адрес в системе → Навигация → Фото подтверждение
    │
    └─── РЕЙС ЗАВЕРШЁН (только после подтверждения сдачи груза + порожнего)
```

### Сводная таблица: что есть / что нужно доработать

| # | Стандарт SFM | Текущий статус | Что нужно |
|---|-------------|---------------|-----------|
| 1 | **Container ID** (идентификация контейнера по полному номеру на всех этапах) | Нет: контейнеры не являются отдельной сущностью, груз описывается только по типу/весу/объёму | **Новый модуль:** сущность «Контейнер» (полный номер, владелец, статус, история операций), обязательная сверка номера при погрузке/выпуске/сдаче, QR/RFID-сканирование, блокировка операций при несовпадении |
| 2 | **Clearance** (цифровые разрешения на выпуск контейнера с терминала) | Нет: нет workflow разрешений от нескольких сторон (СВХ, КТЖ, КДТС, ТХК) | **Новый модуль:** единый статус разрешений для контейнера, цифровые подтверждения от каждой стороны, блокировка заказа машины до получения всех разрешений, автоматическая проверка перед въездом/выездом |
| 3 | **Slot & Queue** (управление очередью и заездом на терминал) | Нет: нет системы слотов, очередей и бронирования времени погрузки | **Новый модуль:** назначение временного слота, электронная очередь, автоперераспределение при опоздании, уведомление крановщика, прозрачная очередь для всех участников |
| 4 | **Loading & Handling** (синхронная погрузка контейнера) | Частично: есть точки погрузки/выгрузки с координатами и отметками прибытия/убытия | Доработать: предварительное назначение машина ↔ контейнер, автоматический расчёт ETA прибытия на терминал, отображение приближения для крановщика, подготовка контейнера заранее, фиксация факта погрузки с временем |
| 5 | **Route & Tracking** (контроль движения и выполнения рейса) | Есть: GPS-трекинг, публичный трекинг, карты, история событий | Доработать: понятные этапы рейса для контейнерных перевозок (в пути → прибыл на склад → выгрузка → выехал → сдача порожнего → завершён), видимость для клиента с ETA и подготовкой склада, контроль отклонений и необоснованных простоев |
| 6 | **Warehouse Delivery** (выгрузка на складе клиента с фиксацией всех этапов) | Частично: есть отметка прибытия/убытия, фото-фиксация | Доработать: фиксация проверки пломб, отдельный статус для комиссионной выдачи, замер времени выгрузки (начало/конец), разделение ответственности за задержки (склад vs водитель), подтверждение сдачи груза |
| 7 | **Empty Container Return** (контроль сдачи порожнего контейнера) | Частично: есть статус AWAITING_RETURN_CONFIRMATION, базовый workflow возврата | Доработать: указание адреса сдачи в системе (не устно), навигация к месту сдачи, фото/цифровое подтверждение сдачи (штамп, фото на территории), привязка сдачи к рейсу, перевозка завершена только после подтверждения сдачи груза + порожнего |
| 8 | **Driver Management** (допуск к рейсу, медосмотр, учёт по водителю) | Частично: есть привязка водитель ↔ ТС, история рейсов, гибкое назначение | Доработать: **медицинское освидетельствование** перед рейсом (цифровой допуск от медработника), блокировка назначения без мед. допуска, фиксация нарушений скорости/маршрута, основа для справедливой мотивации |
| 9 | **Incident & Exception** (инциденты и нестандартные ситуации) | Частично: есть SOS-система, история событий, модуль споров | Доработать: структурированная фиксация инцидентов (повреждённый контейнер, комиссионная выдача, поломка, пробка, ДТП, отсутствие топлива, вскрытие контейнера), категории с привязкой к рейсу, фото/текст подтверждение, разделение ответственности, аналитика |
| 10 | **Fuel & Vehicle Integrity** (контроль топлива, пробега и техсостояния) | Частично: GPS-пробег можно рассчитать по треку, расстояние маршрута фиксируется | **Новый модуль:** контроль пробега только по GPS (не одометр), привязка топлива к рейсу с лимитом, исключение ручного ввода пробега, контроль заправок/сливов, фиксация техсобытий (вскрытие баков, отключение датчиков), аналитика отклонений |
| 11 | **Analytics & Motivation** (отчёты, KPI и справедливая мотивация) | Частично: 26 типов отчётов, ECharts, Excel-экспорт, сравнение исполнителей | Доработать: учёт по типам перевозок (гружёные/порожние/межгород), финансовые показатели на 1 контейнер / 1 км, KPI водителей (рейсы, время, нарушения), автоматический расчёт ЗП/премий/бонусов, прозрачность по ролям |

## 3. ГРУППИРОВКА ДОРАБОТОК ПО БЛОКАМ

### БЛОК А: Контейнер и разрешения (ядро SFM)
- Сущность «Контейнер» (полный номер, владелец, тип 20ft/40ft/HC/рефрижератор, текущий статус, история операций)
- Цифровой профиль контейнера с обязательной сверкой на всех этапах
- QR-код / RFID-сканирование контейнера в мобильном приложении
- Система разрешений: единый статус от СВХ, КТЖ, КДТС, ТХК
- Цифровое подтверждение от каждой стороны с датой/временем
- Блокировка заказа машины без полного пакета разрешений
- Автоматическая проверка при въезде/выезде с терминала

### БЛОК Б: Слоты, очередь и погрузка
- Назначение временного слота (окна) для заезда на терминал
- Электронная очередь с приоритетами (собственный автопарк vs сторонние)
- Автоперераспределение очереди при опоздании (очередь не ждёт)
- Предварительное назначение: машина → контейнер → слот
- Автоматический расчёт ETA прибытия машины на терминал
- Отображение приближения машины для крановщика и терминальных служб
- Подготовка контейнера к погрузке заранее по данным ETA
- Фиксация факта начала/окончания погрузки
- Информирование о готовности терминала и крана

### БЛОК В: Контроль рейса и выгрузка
- Этапы рейса контейнерной перевозки: в пути → прибыл на склад → выгрузка → выехал → сдача порожнего → завершён
- Обновление ETA в реальном времени для клиента
- Видимость для клиента: движение машины, время прибытия, подготовка склада
- Фиксация проверки пломб и состояния контейнера
- Статус комиссионной выдачи (при повреждении)
- Замер времени выгрузки (начало/конец)
- Разделение ответственности за задержки на складе
- Подтверждение сдачи груза клиенту

### БЛОК Г: Сдача порожнего контейнера
- Указание/изменение адреса сдачи в системе
- Навигация водителя к месту сдачи через мобильное приложение
- Фото-подтверждение сдачи (штамп терминала, фото контейнера на площадке)
- Электронная отметка принимающей стороны
- Привязка сдачи порожнего к конкретному рейсу, машине, водителю
- Перевозка завершена только после подтверждения: сдача груза + сдача порожнего

### БЛОК Д: Водители и допуск к рейсу
- Медицинское освидетельствование перед рейсом
- Цифровой медицинский допуск (подтверждение медработником в системе)
- Блокировка назначения на рейс без мед. допуска
- Гибкая привязка водителя к любому автомобилю из автопарка (уже есть, доработать)
- Фиксация нарушений: опоздания, отклонения от маршрута, простои без причины, превышение скорости
- История работы водителя: рейсы, простои, инциденты
- Основа для справедливой мотивации (ЗП, премии, бонусы по данным системы)

### БЛОК Е: Инциденты и исключения
- Структурированные типы инцидентов: повреждённый контейнер, комиссионная выдача, поломка, пробка, ДТП, отсутствие топлива, задержка склада, вскрытие контейнера
- Фиксация инцидента в момент возникновения (водитель/диспетчер)
- Привязка к рейсу, машине, водителю, контейнеру
- Фиксация времени начала и длительности
- Фото и текстовое подтверждение
- Разделение ответственности: инциденты не по вине водителя не влияют на его оценку
- Аналитика инцидентов для управленческих решений

### БЛОК Ж: Топливо, пробег и техсостояние
- Пробег только по GPS (не по одометру)
- Норма топлива рассчитывается автоматически: маршрут + расстояние + тип ТС
- Привязка топлива к конкретному рейсу с лимитом
- Исключение ручного ввода пробега
- Контроль заправок/сливов (время, место, объём, датчики уровня)
- Фиксация техсобытий: вскрытие баков, вмешательство в электронику, отключение датчиков
- Привязка ответственности к водителю и рейсу
- Автоматическое выявление аномалий (нетипичные маршруты, избыточный пробег, повышенный расход)

### БЛОК З: Аналитика, KPI и мотивация
- Единый учёт всех рейсов: машина + водитель + контейнер + клиент + тип перевозки
- Учёт времени: движение, погрузка, выгрузка, простои с причинами
- Финансовые показатели: доходы/расходы на рейс, машину, водителя, включая топливо и простои
- Показатели на единицу: стоимость на 1 контейнер, на 1 км
- Отчёты за любой период: по перевозкам, машинам, водителям, клиентам, типам перевозок
- KPI: количество рейсов, среднее время, простои, нарушения, расход топлива
- Автоматический расчёт ЗП, премий и бонусов по данным системы
- Прозрачность по ролям (водитель, диспетчер, руководитель, клиент видят свои данные)

| **Этап 1** (MVP) | **10–12 недель** | Блоки А + Б: контейнер, разрешения, слоты, очередь, погрузка — ядро терминальной логистики |
| **Этап 2** | **+4–5 недель** | Блоки В + Г: контроль рейса, выгрузка на складе, сдача порожнего — полный цикл перевозки |
| **Этап 3** | **+4–5 недель** | Блоки Д + Е: водители с мед. допуском, инциденты — управление персоналом и исключениями |
| **Этап 4** | **+4–5 недель** | Блоки Ж + З: топливо/пробег, полная аналитика и KPI — экономика и мотивация |

**Общий срок: ~6–7 месяцев** (с учётом параллельной работы и тестирования)

## 5. ИНТЕГРАЦИЯ ИИ

### Где ИИ даёт максимальную пользу для SFM

| Задача | Что делает ИИ | Приоритет |
|--------|--------------|-----------|
| **Прогноз ETA** | ML-модель предсказывает время прибытия машины на терминал/склад с учётом истории, пробок, времени суток, дня недели | Высокий |
| **Оптимизация очереди** | Автоматическое распределение слотов с минимизацией простоев терминала и машин | Высокий |
| **Распознавание контейнеров** | CV-модель для автоматического считывания номера контейнера по фото/видео (OCR) | Высокий |
| **Обнаружение аномалий** | Выявление необычных маршрутов, подозрительных остановок, аномального расхода топлива, вмешательства в датчики | Средний |
| **Прогноз загрузки терминала** | Предсказание потока контейнеров для планирования ресурсов терминала (краны, персонал) | Средний |
| **Рекомендации диспетчеру** | ИИ-ассистент подсказывает оптимальное назначение машин, перераспределение очереди, реакцию на инциденты | Средний |
| **Анализ причин задержек** | Кластеризация и выявление системных проблем (конкретный терминал, склад, время суток, водитель) | Средний |
| **Прогноз поломок** | Предиктивное ТО по данным GPS, пробега, расхода топлива и истории ремонтов | Низкий (след. этап) |
| **Оценка водителей** | ML-скоринг водителей по комплексу показателей для справедливой мотивации | Низкий |

### Рекомендуемый подход к ИИ

**Фаза 1 — Сбор данных (параллельно с Этапами 1-2):**
- Накопление данных: рейсы, контейнеры, слоты, очереди, ETA факт/план, инциденты, топливо
- Структурирование данных для ML-пайплайна
- Интеграция данных GPS с историей операций

**Фаза 2 — Базовые модели (Этап 3):**
- ML-прогноз ETA (XGBoost/LightGBM на табличных данных)
- Детекция аномалий (Isolation Forest для GPS-треков и расхода топлива)
- OCR-распознавание номера контейнера (YOLO + Tesseract или PaddleOCR)
- Отдельный Python-микросервис для ML-моделей

**Фаза 3 — Продвинутая аналитика (после запуска):**
- Оптимизация очередей терминала (OR-Tools / CP-SAT solver)
- LLM-ассистент диспетчера (Claude API — анализ ситуации и рекомендации на русском языке)
- Предиктивное ТО
- ML-скоринг водителей

### Рекомендуемые ИИ-решения

| Задача | Технология | Примечание |
|--------|-----------|------------|
| Прогноз ETA | Python + XGBoost / LightGBM | Лучшие результаты для табличных данных с GPS-фичами |
| OCR контейнеров | YOLO v8 + PaddleOCR | Специализированные модели для container number recognition |
| Обнаружение аномалий | Isolation Forest / DBSCAN | GPS-треки, расход топлива, время простоев |
| Оптимизация очередей | Google OR-Tools / CP-SAT | Проверенные решения для задач scheduling |
| ИИ-ассистент диспетчера | Claude API | Анализ ситуации и рекомендации на русском языке |
| Прогноз загрузки | Prophet / ARIMA + XGBoost | Временные ряды + сезонность |
| Предиктивное ТО | TensorFlow + временные ряды | После накопления данных по ТО |

### Сроки по ИИ

| Фаза | Срок | Ресурсы |
|------|------|---------|
| Фаза 1: сбор данных | Параллельно (0 доп. недель) | Архитектор данных |
| Фаза 2: базовые ML-модели + OCR | 8–10 недель | 1 ML-инженер + 1 CV-инженер |
| Фаза 3: продвинутый ИИ | 8–12 недель | 1 ML-инженер + 1 бэкенд |

## 6. ИТОГОВАЯ КАРТИНА

### Что уже можно переиспользовать из Coube (экономия ~50% времени):
- GPS-трекинг транспорта в реальном времени (бэк + мобилка + фронт)
- Публичный трекинг по ссылке с ETA и маршрутом
- Карты Яндекс с маршрутами и анимацией движения
- Push/SMS/WhatsApp/Telegram уведомления
- Управление водителями, ТС, сотрудниками
- Гибкая привязка водитель ↔ автомобиль
- Система ролей и авторизации (Keycloak)
- Генерация документов (PDF/Excel)
- Электронная подпись (NCALayer, eGov Mobile)
- Мобильное приложение водителя (основа: GPS, заказы, отметки, фото, SOS)
- Система отчётов с ECharts (каркас для 26 типов отчётов)
- PostGIS и геопространственная БД
- Фото-фиксация и работа с файлами (MinIO)
- История событий рейса (19 типов)

### Что нужно разработать с нуля:
- Сущность «Контейнер» с цифровым профилем и обязательной сверкой
- Модуль разрешений (clearance) от нескольких сторон
- Система слотов и электронной очереди на терминале
- QR/RFID сканирование контейнеров в мобилке
- Медицинский допуск водителя
- Структурированные инциденты с категориями
- Контроль топлива и GPS-пробега
- Учёт и подтверждение сдачи порожнего контейнера
- Новые типы отчётов: по контейнерам, терминалам, топливу, KPI водителей
- Автоматический расчёт ЗП/премий/бонусов
- ИИ-модели (отдельный этап)

### Общая оценка

| Параметр | Значение |
|----------|----------|
| Доля готового функционала | ~35-40% |
| Доля доработок существующего | ~25-30% |
| Доля нового функционала | ~30-35% |
| Срок MVP (контейнеры + терминал + очередь) | 10-12 недель |
| Срок полного внедрения (без ИИ) | ~6-7 месяцев |
| Срок с базовым ИИ (ETA + OCR + аномалии) | +8-10 недель |
| Минимальная команда | 5-6 человек |

### Ключевые отличия SFM от стандартного Coube

| Аспект | Текущий Coube | SFM |
|--------|--------------|-----|
| Объект перевозки | Абстрактный груз | Контейнер с полным номером и профилем |
| Терминальные операции | Нет | Слоты, очередь, разрешения, погрузка |
| Контроль допуска | Роли и авторизация | + медосмотр водителя, + разрешения на контейнер |
| Завершение рейса | Статус FINISHED | Сдача груза + сдача порожнего + подтверждение |
| Инциденты | SOS-сигнал | 8+ категорий с фото, временем, ответственностью |
| Топливо/пробег | Нет | GPS-пробег, нормативы, контроль заправок/сливов |
| Мотивация | Базовые отчёты | Автоматический расчёт ЗП и KPI из данных системы |
| Клиентская видимость | Публичный трекинг | + ETA для подготовки склада, + статус разрешений |
