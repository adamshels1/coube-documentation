## 🔑 Основные принципы MVP

### 1. Переиспользуем существующее (95% кода)

✅ **Entity:**
- `Transportation` - основная сущность заявки (добавляем только 3 поля)
- `TransportationRouteHistory` - версии маршрута (не трогаем)
- `CargoLoadingHistory` - точки маршрута = точки доставки! (добавляем 3 поля)
- `Employee` - водитель/курьер (одна роль `DRIVER`)

✅ **Controllers:**
- `DriverController` - 95% API для курьера уже готов
- Добавляем только 1 метод для обновления статуса заказа

✅ **Services:**
- `DriverService` - вся логика работы с заявками
- `TransportationService` - создание заявок
- `TransportationRouteService` - управление маршрутами
- `FileService` - загрузка фото
- `NotificationService` - уведомления
- `DriverLocationService` - геолокация

### 2. Минимум новых компонентов

🆕 **Новые таблицы (только 2!):**
- `courier_route_order` - заказы в точках маршрута
- `courier_integration_log` - логи интеграций с TEEZ

🆕 **Новые сервисы (всего 4!):**
- `CourierIntegrationService` - импорт маршрутных листов от TEEZ
- `CourierWaybillService` - управление маршрутами для логиста (редактирование, валидация)
- `CourierOrderService` - управление заказами курьера
- `CourierResultsService` - отправка результатов в TEEZ

🆕 **Новые контроллеры (всего 2!):**
- `CourierIntegrationController` - API для TEEZ (импорт маршрутных листов)
- `CourierWaybillController` - API для логиста (веб-интерфейс)

### 3. Одна роль для водителей и курьеров

**НЕТ новой роли `COURIER`!** Используется существующая `DRIVER`.

Различие по типу Transportation:
```java
// Обычный водитель
transportation.setTransportationType(TransportationType.FLT);

// Курьер
transportation.setTransportationType(TransportationType.COURIER_DELIVERY);
```

## 🚀 Основной функционал MVP

### Что включено:

✅ **Импорт маршрутных листов от TEEZ**
- REST API для приема данных
- Создание Transportation с типом COURIER_DELIVERY
- Парсинг точек доставки и заказов

✅ **Мобильное приложение для курьера**
- Список заявок (через DriverController)
- Прием/отклонение заявки
- Отметка точек маршрута (arrival/departure)
- Обновление статусов заказов
- Загрузка фото подтверждений

✅ **Веб-интерфейс для логиста (по ТЗ!)**
- Список маршрутных листов
- Просмотр деталей маршрута
- **Редактирование маршрута** (добавление/удаление точек)
- **Сохранение с валидацией** (геокодирование, проверки, статус SIGNED_CUSTOMER)
- Назначение курьера на маршрут
- Мониторинг выполнения

✅ **Отправка результатов в TEEZ**
- Синхронная отправка при закрытии маршрута
- Логирование всех интеграций

✅ **API Key аутентификация (упрощенная)**
- Статический ключ в environment variable
- Простой Security Filter без БД
- Безопасно и быстро в реализации (2-4 часа)

### Что НЕ включено (можно добавить позже):

❌ Асинхронная отправка результатов с retry  
❌ Управление складами через admin UI  
❌ Детальная аналитика и отчеты  
❌ История изменений маршрутов  
❌ Поддержка нескольких маркетплейсов (только TEEZ в MVP)  
❌ Полная система управления API ключами (БД + Admin UI)  
❌ IP whitelist и rate limiting для API  
❌ BCrypt хеширование API ключей (простое сравнение строк в MVP)

## 🔄 Бизнес-процесс

```
1. TEEZ → Coube: POST /api/v1/integration/waybills
   ├── Импорт маршрутного листа
   ├── Создание Transportation с типом COURIER_DELIVERY
   ├── Создание рейса через TransportationRouteService.createInitialRoute()
   ├── Создание точек маршрута (CargoLoadingHistory)
   ├── Создание заказов (CourierRouteOrder)
   └── Статус: FORMING (черновик)

2. Логист (через веб) - РЕДАКТИРОВАНИЕ:
   ├── GET /api/v1/executor/transportations → список маршрутов (фильтр COURIER_DELIVERY)
   ├── GET /api/v1/executor/transportations/{id} → детали маршрута
   ├── PUT /api/v1/executor/transportations/{id}/route → редактирование маршрута
   ├── POST /api/v1/executor/transportations/{id}/save → сохранение с валидацией
   ├── Статус: FORMING → SIGNED_CUSTOMER (сохранен после валидации)
   └── Валидация: геокодирование адресов, временные окна, последняя точка = склад

3. Логист - НАЗНАЧЕНИЕ КУРЬЕРА:
   ├── POST /api/v1/executor/{id}/assign-courier
   ├── Выбор курьера (без ТС или с ТС)
   └── Статус: SIGNED_CUSTOMER → WAITING_DRIVER_CONFIRMATION

4. Курьер (через мобильное приложение):
   ├── GET /api/v1/driver/orders → видит свои маршруты
   ├── PUT /orders/{id}/accept → принимает маршрут
   ├── PUT /orders/{id}/start → начинает выполнение
   ├── Для каждой точки:
   │   ├── PUT /orders/{id}/arrival → прибытие
   │   ├── Для каждого заказа в точке:
   │   │   ├── PUT /orders/{id}/courier-orders/{orderId}/status
   │   │   ├── Загрузка фото (если требуется)
   │   │   └── Статус: DELIVERED / RETURNED / NOT_DELIVERED
   │   └── PUT /orders/{id}/departure → отбытие
   └── Статус: WAITING_DRIVER_CONFIRMATION → DRIVER_ACCEPTED → ON_THE_WAY → FINISHED

5. Автоматическая отправка результатов:
   ├── При переходе в статус FINISHED
   └── Coube → TEEZ автоматически

6. Coube → TEEZ: POST {teez_api_url}/api/waybill/results
   ├── Отправка результатов по всем заказам (автоматически при закрытии)
   ├── Логирование интеграции
   └── В случае ошибки - показать логисту
```

## 🔔 Последнее обновление (2025-01-28)

### ⭐ Задача 26: Автоматическое назначение курьера при импорте заявки

**Файл**: `26-auto-courier-assignment-on-import.md`

**Инициатор**: TEEZ

**Проблема**:
- TEEZ хочет отправлять заявку с уже назначенным курьером
- Заявка должна сразу попадать в мобильное приложение курьера без действий логиста

**Решение**:
1. **Новый метод**: `GET /api/v1/integration/couriers` - список курьеров для TEEZ
2. **Доработка импорта**: добавить опциональное поле `courierPhone` в `POST /api/v1/integration/waybills`
3. **Автоматическое назначение**: если переданы все данные (courierPhone + полные данные точек), заявка автоматически валидируется и назначается курьеру

**Логика**:
- ✅ Передан `courierPhone` + полные данные точек (адрес + координаты) → автоматическая валидация + назначение → статус `WAITING_DRIVER_CONFIRMATION`
- ⚠️ Не передан `courierPhone` или неполные данные → статус `IMPORTED` (как раньше)
- ❌ Курьер не найден/неактивен → HTTP 404, заявка НЕ создается
- ❌ Курьер занят → HTTP 409, заявка НЕ создается

**Обратная совместимость**: Полностью совместимо, если не передавать `courierPhone` - работает как раньше

**Время реализации**: 12 часов (1.5 дня)
**Приоритет**: HIGH - запрос от TEEZ для упрощения процесса

### ⚠️ Задача 25: CRITICAL - Исправление бага с преждевременным завершением маршрута

**Файл**: `25-fix-departure-last-point-bug.md`

**Проблема**:
- 🔴 При отъезде со склада (первой точки) маршрут сразу завершается (статус `FINISHED`)
- 🔴 Курьер не может доехать до точки доставки и выполнить заказы
- 🔴 Блокирует весь функционал курьерской доставки

**Баг в коде**:
- Файл: `DriverService.java:442`
- Условие: `if (nextOrderNum == transportation.getCargoLoadings().size())`
- Ошибка: Проверка срабатывает когда водитель **едет К последней точке**, а не когда **отъезжает С последней точки**

**Решение**:
```java
// Было:
if (nextOrderNum == transportation.getCargoLoadings().size()) {

// Стало:
if (nextOrderNum > transportation.getCargoLoadings().size()) {
```

**Дополнительно**:
- ✅ Unit и Integration тесты (5 unit + 2 integration)
- ✅ Manual тестирование с мобильным приложением (5 сценариев)

**Время реализации**: 2.5-3 часа (упрощенная логика)
**Приоритет**: 🔴 CRITICAL - требует немедленного исправления
**Severity**: Блокирует весь функционал курьерской доставки

**Упрощение логики**:
- Убрана проверка статусов заказов на последней точке
- Тип последней точки (`isCourierWarehouse`) не влияет на логику завершения
- Единственная проверка: наличие возвратов в маршруте
  - Есть возвраты → `AWAITING_RETURN_CONFIRMATION`
  - Нет возвратов → `FINISHED`
