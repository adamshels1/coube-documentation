# Reviews Module - Модуль отзывов

## Описание

Модуль для управления отзывами между исполнителями и заказчиками в системе Coube. Позволяет организациям оценивать друг друга после завершения перевозок, а суперадминам модерировать эти отзывы.

- Версия: **MVP** (минимальный функционал)
- Автор миграций БД: Back-N-Trick

## Структура БД

### Схема: `reviews`

#### Таблицы:
1. **reviews.reviews** - основная таблица отзывов
   - Отзывы от исполнителей к заказчикам и наоборот
   - Рейтинги (1-5 звезд) с детализацией по критериям
   - Статусы (MVP): `pending`, `approved`, `rejected`
   - Поля модерации: moderated_at, moderator_id, **moderation_reason** (обязательно при reject)

2. **reviews.review_flags** - жалобы на отзывы
   - Механизм жалоб от организаций
   - История флагов на каждый отзыв

3. **reviews.user_reputation** - кэш репутации организаций
   - Средний рейтинг
   - Количество отзывов
   - Распределение по звездам (1-5)

## Задачи

### TASK-1: API для модерации отзывов суперадмином (MVP)
**Статус**: 🔴 To Do
**Приоритет**: High
**Story Points**: 5

**Описание**: Разработка минимального REST API для модерации отзывов суперадмином

**Основные endpoints (MVP)**:
- GET `/api/v1/admin/reviews/moderation` - список отзывов для модерации
- GET `/api/v1/admin/reviews/{id}` - детали отзыва с репутацией организаций
- PUT `/api/v1/admin/reviews/{id}/approve` - одобрение отзыва
- PUT `/api/v1/admin/reviews/{id}/reject` - отклонение отзыва с **обязательной причиной**

**Ключевые особенности**:
- ✅ Причина отклонения (`moderation_reason`) - обязательное поле (10-500 символов)
- ✅ Автоматический пересчет репутации организаций
- ✅ Статусы: `pending`, `approved`, `rejected` (flagged не используется в MVP)

**Файл**: [TASK-1-reviews-moderation-api.md](./TASK-1-reviews-moderation-api.md)

## Роли и доступы

- **SUPER_ADMIN** - полный доступ к модерации отзывов
- **EXECUTOR** - может оставлять отзывы заказчикам (будет в TASK-6)
- **CUSTOMER** - может оставлять отзывы исполнителям (будет в TASK-6)

## Бизнес-процесс (MVP)

1. Перевозка завершается (статус COMPLETED)
2. Исполнитель и заказчик могут оставить отзывы друг на друга
3. Отзыв создается со статусом `pending`
4. Суперадмин проверяет отзыв и:
   - **Одобряет** (approved) → отзыв становится публичным, учитывается в репутации
   - **Отклоняет** (rejected) с обязательной причиной → отзыв скрывается, не учитывается в репутации
5. Репутация организации пересчитывается автоматически
6. ~~Организация получает уведомление~~ (отложено на TASK-6)
