# Анализ: Добавление FTL-флоу для курьерской доставки

**Дата:** 3 апреля 2026  
**Задача:** Дать возможность заказчику создавать курьерскую заявку вручную (как FTL), чтобы перевозчик её принял, подписал договор и исполнил.

## 2. FTL-ФЛОУ (КАК ЕСТЬ)

### Полный цикл

```
Заказчик создаёт заявку (FORMING)
  ↓ Публикует
Заявка ждёт откликов (CREATED)
  ↓ Перевозчики откликаются
Заказчик выбирает перевозчика (WAITING_CUSTOMER_DECISION)
  ↓ Подписывает договор EDS
Договор подписан (SIGNED_CUSTOMER)
  ↓ Перевозчик назначает водителя
Ожидание водителя (WAITING_DRIVER_RESPONSE → WAITING_DRIVER_CONFIRMATION)
  ↓ Водитель принимает
Водитель назначен (DRIVER_ACCEPTED)
  ↓ Начало перевозки
В пути (ON_THE_WAY)
  ↓
[опционально: SOS]
  ↓
Завершено (FINISHED)
  ↓
Счет + Акт + [Факторинг]
```

### Статусы Transportation

| Статус | Описание |
|--------|----------|
| `FORMING` | Черновик заявки |
| `CREATED` | Опубликована, ожидает перевозчиков |
| `WAITING_CUSTOMER_DECISION` | Есть отклики, заказчик выбирает |
| `SIGNED_CUSTOMER` | Договор подписан |
| `WAITING_DRIVER_RESPONSE` | Перевозчик должен назначить водителя |
| `WAITING_DRIVER_CONFIRMATION` | Водитель получил задачу |
| `DRIVER_ACCEPTED` | Водитель принял |
| `ON_THE_WAY` | В пути |
| `SOS` | Критическая ситуация |
| `AWAITING_RETURN_CONFIRMATION` | Ожидает подтверждение возврата |
| `FINISHED` | Завершено |
| `CANCELED` | Отменено |
| `IMPORTED` | Загружена из внешней системы (курьерка) |
| `VALIDATED` | Провалидирована логистом (курьерка) |

### Ключевые эндпоинты FTL

**Customer API:**
```
POST /api/v1/customer/transportations          — Создание заявки
PUT  /api/v1/customer/transportations/{id}     — Обновление
POST /api/v1/customer/transportations/{id}/publish  — Публикация
POST /api/v1/customer/agreements/{id}/sign     — Подпись EDS
```

**Executor API:**
```
GET  /api/v1/executor/transportations          — Список доступных заявок
POST /api/v1/executor/counter-offer/{id}       — Встречное предложение по цене
POST /api/v1/executor/accept-price/{id}        — Принятие цены
POST /api/v1/executor/agreements/{id}/accept   — Принятие договора
POST /api/v1/executor/{id}/assign-courier      — Назначение водителя
```

**Driver API:**
```
GET  /api/v1/driver/orders                     — Список назначенных заявок
PUT  /api/v1/driver/orders/{id}/accept         — Принять
PUT  /api/v1/driver/orders/{id}/reject         — Отклонить
PUT  /api/v1/driver/orders/{id}/start          — Начать
PUT  /api/v1/driver/orders/{id}/arrival        — Прибытие на точку
PUT  /api/v1/driver/orders/{id}/departure      — Отбытие с точки
POST /api/v1/driver/orders/{id}/sos            — SOS
```

## 3. КУРЬЕРСКАЯ ДОСТАВКА (КАК ЕСТЬ)

### Текущий флоу

```
Импорт из внешней системы (TEEZ_PVZ, KASPI, WILDBERRIES, OZON)
  ↓ POST /api/v1/integration/waybills
Маршрутный лист создан (IMPORTED)
  ↓ Логист валидирует
Провалидирован (VALIDATED)
  ↓ Логист назначает курьера
Ожидание подтверждения (WAITING_DRIVER_CONFIRMATION)
  ↓ Курьер принимает
Курьер назначен (DRIVER_ACCEPTED)
  ↓
В пути (ON_THE_WAY)
  ↓ Для каждого заказа в точке: PENDING → DELIVERED/RETURNED/NOT_DELIVERED
Завершено (FINISHED)
  ↓ Результаты уходят обратно в TEEZ
```

### Отличия от FTL

| Параметр | FTL | Курьерская (сейчас) |
|----------|-----|----------------------|
| Создание | Заказчик вручную | Импорт из внешней системы |
| Маршрут | 1-2 точки | 5-50+ точек |
| Заказы | 1 заказ на перевозку | Много заказов в каждой точке |
| Перевозчик | Сторонний (выбирается из откликов) | Заказчик сам назначает своего курьера |
| Договор EDS | Да | Нет |
| Статус заказа | Только транспортировка | Детальный (delivered, returned, etc.) |
| SMS/Фото | Опционально | Обязательно в большинстве точек |
| Факторинг | Да | Не реализован |

### Таблицы БД для курьерской доставки

| Таблица | Назначение |
|---------|-----------|
| `applications.transportation` | Основная заявка (маршрутный лист) |
| `gis.transportation_route_history` | Версия маршрута |
| `gis.cargo_loading_history` | Точки доставки (5-50+) |
| `applications.courier_route_order` | Заказы в точках (несколько на точку) |
| `applications.courier_integration_log` | Логи вызовов внешних систем |

### Статусы CourierRouteOrder

```
PENDING → IN_PROGRESS → DELIVERED
                      → RETURNED
                      → PARTIALLY_RETURNED
                      → NOT_DELIVERED (с причиной: CUSTOMER_NOT_AVAILABLE / CUSTOMER_POSTPONED / FORCE_MAJEURE)
```

## 4. ЗАДАЧА: ЧТО НУЖНО ДОБАВИТЬ

### Суть изменения

Добавить возможность для курьерской доставки работать по **FTL-флоу**:
- Заказчик **вручную создаёт** курьерскую заявку (не только импорт)
- Заявка **публикуется** — перевозчики видят её
- Перевозчик **откликается** (предлагает цену)
- Заказчик **выбирает** перевозчика, **подписывается договор EDS**
- Перевозчик **назначает своего водителя/курьера**
- Курьер **принимает и исполняет** маршрут
- **Документы**: счет, акт, реестр

### Новый флоу (целевой)

```
Заказчик создаёт курьерскую заявку вручную (FORMING)
  ↓ Заполняет маршрут (5-50+ точек), грузы, требования
Публикует (CREATED)
  ↓ Перевозчики-курьеры видят заявку
Отклики перевозчиков (WAITING_CUSTOMER_DECISION)
  ↓ Заказчик выбирает перевозчика
Подписание договора EDS (SIGNED_CUSTOMER)
  ↓ Перевозчик назначает курьера
Курьер назначен (WAITING_DRIVER_CONFIRMATION → DRIVER_ACCEPTED)
  ↓
В пути (ON_THE_WAY)
  ↓ Детальный трекинг по заказам в точках
Завершено (FINISHED)
  ↓ Счет + Акт + Реестр
```

## 5. ПОДВОДНЫЕ КАМНИ

### 5.1. Архитектурные

#### Переиспользование CargoLoadingHistory
- **Проблема:** `CargoLoadingHistory` спроектирован для FTL (1-2 точки), для курьера нужно 5-50+
- **Статус:** Таблица уже поддерживает множество точек, но нет UI для массового создания
- **Риск:** Производительность при 50+ точках (нужны индексы)

#### Два уровня статусов
- **Проблема:** Есть `TransportationStatus` (маршрут целиком) и `CourierOrderStatus` (каждый заказ в точке)
- **Риск:** Путаница в логике перехода — когда маршрут `FINISHED`, но отдельные заказы ещё `NOT_DELIVERED`
- **Решение:** Чёткие правила: маршрут `FINISHED` когда **все точки пройдены**, независимо от статуса заказов

#### Роли DRIVER и LOGISTICIAN
- **Проблема:** Текущий курьерский флоу управляет логист (LOGISTICIAN), а FTL — executor/driver
- **Риск:** Возникнет конфликт ролей — кто управляет курьерской заявкой с FTL-флоу?
- **Решение:** Нужно определить: логист курьерской компании = исполняет роль EXECUTOR для курьерки

#### Маршрут с множеством точек + EDS
- **Проблема:** В FTL при изменении маршрута требуется **переподписание** обеими сторонами (EDS)
- **Риск:** Для 50-точечного маршрута переподписание при каждом изменении — нереалистично
- **Решение:** Либо упрощённая схема (SMS-подтверждение вместо EDS для изменений), либо оригинальный маршрут подписывается, изменения согласовываются иначе

### 5.2. Бизнесовые

#### Договор для курьерки
- **Проблема:** В FTL договор один на всю перевозку. В курьерке на маршрутном листе много заказчиков (разные отправители/получатели)
- **Вопрос:** Договор будет между **организацией-заказчиком** (кто создаёт заявку) и **перевозчиком**?
- **Ответ нужен:** Подтвердить модель — один договор на весь маршрут

#### Ценообразование
- **Проблема:** В FTL цена за всю перевозку. В курьерке цена может быть:
  - За маршрут (фиксировано)
  - За доставленный заказ
  - За километр
- **Риск:** Нужна новая модель ценообразования или адаптация существующей `TransportationCost`

#### Частичное исполнение
- **Проблема:** Курьер не доставил 20% заказов → как считается оплата?
- **Риск:** Нужна логика пересчёта стоимости при `NOT_DELIVERED` заказах

#### Факторинг для курьерки
- **Проблема:** `FactoringPayoutRequest` привязан к FTL-перевозкам
- **Риск:** Для курьерки с FTL-флоу нужно расширить факторинг
- **Приоритет:** Низкий — можно добавить позже

### 5.3. Технические

#### API для создания курьерской заявки
- **Нужно добавить:** Endpoint `POST /api/v1/customer/transportations` должен поддерживать `transportationType=COURIER_DELIVERY` с массовым вводом точек
- **Текущее:** Работает для FTL, но нет UI/логики для ввода 50+ точек

#### Мобильное приложение
- **Проблема:** В мобилке есть экраны для курьерки (CourierOrderScreen) и для FTL (TransportationsDetailsScreen)
- **Риск:** Для нового флоу нужно убедиться, что экраны корректно обрабатывают курьерку с FTL-статусами

#### Уведомления
- **Нужно добавить:** События уведомлений для:
  - Публикация курьерской заявки → перевозчикам
  - Отклик перевозчика → заказчику
  - Подписание договора → обеим сторонам
  - Назначение курьера → курьеру

#### Производительность
- **Риск:** Маршрут с 50+ точками → медленная загрузка
- **Решение:**
  ```sql
  CREATE INDEX idx_cargo_loading_route_action ON gis.cargo_loading_history(route_history_id, action);
  CREATE INDEX idx_cargo_loading_order_num ON gis.cargo_loading_history(route_history_id, order_num);
  ```
