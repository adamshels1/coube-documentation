# Анализ и предложение по интеграции COUBE с ИС ЭСФ

> **Дата:** 2026-03-16
> **Статус:** Предварительный анализ
> **Вопрос:** Можем ли мы реализовать интеграцию с ИС ЭСФ для пользователей в финансовых документах?

---

## Вывод: ДА, реализация возможна

Интеграция с ИС ЭСФ технически осуществима в рамках существующей архитектуры COUBE. Ниже — детальный анализ, что уже есть, что нужно сделать и как это должно работать.

---

## 1. Что такое ИС ЭСФ

ИС ЭСФ — государственная информационная система КГД МФ РК для электронного обмена счетами-фактурами между участниками рынка Казахстана.

**API:** SOAP web-services (WSDL) на `https://esf.gov.kz:8443/esf-web/`
**Аутентификация:** Сессионная, через ЭЦП (сертификат НУЦ РК)
**SDK:** Java SDK v4.0.0, содержит:
- `esf-client` — отправка ЭСФ
- `vstore-client` — Виртуальный склад (ВС)
- `vstore-model` — модели данных

## 2. Что уже есть в COUBE

### Существующие компоненты (используемые для ЭСФ):

| Компонент | Расположение | Релевантность |
|-----------|-------------|---------------|
| Invoice entity | `invoice/entity/Invoice.java` | Основа для ЭСФ — уже содержит суммы, НДС, контрагентов |
| Цифровая подпись (Калкан) | `signature/` | Подписание XML документов ЭЦП НУЦ РК |
| Organization (с БИН) | `organization/` | Данные продавца и покупателя |
| Transportation (перевозки) | `applications/entity/Transportation.java` | Основание для выставления счёта |
| DocumentStatus | `invoice/entity/DocumentStatus` | Статусная модель уже есть |
| InvoiceDocumentAdapter | `signature/adapter/InvoiceDocumentAdapter.java` | Адаптер для подписания документов |

### Что это означает:
- Инфраструктура подписания **уже готова** (Kalkan/НУЦ РК)
- Модель счёта (Invoice) содержит **все необходимые поля**: суммы с/без НДС, контрагенты, даты
- Статусная модель документов **уже существует**

## 3. Что нужно реализовать

### 3.1. Основной сервис отправки ЭСФ

**Новый модуль:** `coube-backend/src/main/java/kz/coube/backend/esf/`

```
esf/
  ├── client/           — SOAP клиент ИС ЭСФ (сессия, отправка)
  ├── mapper/           — Invoice → ESF XML маппер
  ├── service/          — бизнес-логика
  ├── dto/              — DTO для API
  ├── entity/           — EsfDocument (хранение статусов)
  └── scheduler/        — фоновая синхронизация статусов
```

### 3.2. Маппинг данных COUBE → ЭСФ

| Поле ЭСФ | Источник в COUBE |
|---------|-----------------|
| seller.tin | `Organization.bin` (исполнитель) |
| customer.tin | `Organization.bin` (заказчик) |
| invoiceNum | `Invoice.invoiceNumber` |
| invoiceDate | `Invoice.documentDate` |
| turnoverDate | `Transportation.completedAt` |
| product.name | Услуги перевозки (из заявки) |
| product.unitPrice | Стоимость перевозки |
| product.ndsRate | 12% (стандартная ставка НДС РК) |
| product.ndsAmount | `Invoice.totalVatAmount` |
| product.priceWithTax | `Invoice.totalAmountWithVat` |

### 3.3. Авторизация и сессия

```
Шаг 1: Открыть сессию через SessionService (SOAP)
        → передать подписанный XML с сертификатом ЭЦП
Шаг 2: Получить sessionId
Шаг 3: Использовать sessionId во всех последующих запросах
Шаг 4: Закрыть сессию по завершении или таймауту
```

Используется существующая инфраструктура Kalkan в COUBE для подписания.

### 3.4. Жизненный цикл ЭСФ в COUBE

```
Счёт подписан в COUBE
        ↓
[Новое] Формирование XML ЭСФ
        ↓
[Новое] Подписание XML (Kalkan ЭЦП исполнителя)
        ↓
[Новое] Отправка в ИС ЭСФ через API
        ↓
[Новое] Получение регистрационного номера ЭСФ
        ↓
[Новое] Сохранение статуса (SENT → DELIVERED/DECLINED)
        ↓
[Новое] Отображение статуса пользователю в фин. документах
```

### 3.5. Расширение таблицы Invoice (или новая таблица)

```sql
-- Новая таблица или добавление полей в applications.invoices
ALTER TABLE applications.invoices ADD COLUMN esf_status VARCHAR(50);
ALTER TABLE applications.invoices ADD COLUMN esf_registration_number VARCHAR(100);
ALTER TABLE applications.invoices ADD COLUMN esf_sent_at TIMESTAMP;
ALTER TABLE applications.invoices ADD COLUMN esf_last_sync_at TIMESTAMP;
ALTER TABLE applications.invoices ADD COLUMN esf_error_message TEXT;

-- Или отдельная таблица для истории
CREATE TABLE applications.esf_documents (
    id BIGSERIAL PRIMARY KEY,
    invoice_id BIGINT NOT NULL REFERENCES applications.invoices(id),
    registration_number VARCHAR(100),
    status VARCHAR(50) NOT NULL,   -- DRAFT, SENT, DELIVERED, DECLINED, CANCELLED
    sent_at TIMESTAMP,
    xml_body TEXT,
    error_message TEXT,
    created_at TIMESTAMP DEFAULT now(),
    updated_at TIMESTAMP DEFAULT now()
);
```

## 4. Основные веб-сервисы ИС ЭСФ для интеграции

| Сервис | Метод | Назначение |
|--------|-------|-----------|
| SessionService | `createSession` | Открыть сессию |
| SessionService | `closeSession` | Закрыть сессию |
| InvoiceService | `sendInvoice` | Отправить ЭСФ |
| InvoiceService | `queryInvoice` | Получить статус ЭСФ |
| InvoiceService | `queryInvoiceList` | Список входящих/исходящих ЭСФ |
| InvoiceService | `cancelInvoice` | Отозвать ЭСФ |
| VstoreVersionService | `getApiVersion` | Проверить версию API |
| VstoreVersionService | `getErrorCodes` | Получить коды ошибок |

**Endpoint:** `https://esf.gov.kz:8443/esf-web/ws/api1/` (production)
**Test endpoint:** `https://esf.gov.kz:8080/esf-web/ws/api1/`

## 5. Интеграция с Виртуальным складом (ВС)

Виртуальный склад — дополнительная функциональность ИС ЭСФ для отслеживания движения товаров. Для логистической платформы (COUBE — услуги перевозки) **ВС не является обязательным** на первом этапе.

Рекомендуется оставить на 2-й фазе, если клиенты начнут выставлять ЭСФ на товары через COUBE.

---

## 6. Что увидят пользователи (UX)

### В разделе "Финансовые документы":

1. **Кнопка "Выставить ЭСФ"** — появляется на подписанном счёте
2. **Статус ЭСФ** — рядом со счётом: `Не отправлен | Отправлен | Принят | Отклонён`
3. **Регистрационный номер ЭСФ** — отображается после успешной отправки
4. **Кнопка "Отозвать ЭСФ"** — для отзыва в случае ошибки
5. **Входящие ЭСФ** — вкладка для получения счетов от поставщиков

## 7. Технические требования для запуска

| Требование | Статус | Действие |
|-----------|--------|---------|
| Сертификат ЭЦП НУЦ РК (юр. лицо) | Нужно получить | Регистрация в НУЦ РК |
| Регистрация как интеграционный клиент | Нужно | Заявка в КГД через esfsd@kgd.minfin.gov.kz |
| Подключение к тестовой среде | Нужно | После регистрации |
| ESF SDK v4.0.0 (Java) | Есть в SDK | Добавить в build.gradle |
| Kalkan (для подписания) | Уже есть в backend | Переиспользовать |

### Зависимости для build.gradle:
```groovy
// Из esf-sdk-2025 (установить локально через mvninstall.sh)
```

## 8. Оценка рисков

| Риск | Вероятность | Митигация |
|------|------------|-----------|
| Изменения в API ИС ЭСФ | Средняя | Версионирование, мониторинг обновлений |
| Недоступность ИС ЭСФ | Средняя | Очередь повторных попыток (retry queue) |
| Истечение сертификата ЭЦП | Средняя | Оповещения за 30 дней до истечения |
| Блокировка за частые запросы | Низкая | Не более 60 запросов/мин (лимит ИС ЭСФ) |
| Отклонение ЭСФ (ошибка данных) | Средняя | Валидация перед отправкой, уведомление |

## 10. Итоговая рекомендация

**Реализовать в Фазе 1** следующий минимальный сценарий:

> Пользователь (исполнитель) подписывает счёт в COUBE → нажимает "Выставить ЭСФ" → COUBE автоматически формирует XML, подписывает ЭЦП и отправляет в ИС ЭСФ → пользователь видит статус и регистрационный номер ЭСФ.

Это закрывает ключевую бизнес-потребность (обязательность ЭСФ для плательщиков НДС в РК) и опирается на уже существующую инфраструктуру COUBE (Kalkan, Invoice, Organization).
