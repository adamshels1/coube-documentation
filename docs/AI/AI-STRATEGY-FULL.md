# AI-стратегия платформы Coube — Полный документ

## Обзор

Данный документ описывает стратегию внедрения искусственного интеллекта в платформу Coube. Цель — максимальная автоматизация, аналитика данных и контроль перевозок.

## Текущее состояние

Платформа уже собирает богатые данные для ИИ:
- GPS-треки водителей (координаты, скорость, направление, точность — каждые 30 сек)
- 14-мерная KYC-скоринговая модель
- Полная история перевозок с ценами, маршрутами, статусами
- Данные по загрузке/разгрузке с таймстемпами и геокоординатами
- Уведомления с метриками доставки и вовлечённости
- Финансовые данные (оплаты, задержки, факторинг)

**ИИ-модели в продакшене пока отсутствуют** — вся аналитика ручная, matching на правилах.

## Приоритеты

- **P0** — Критические (максимальный эффект для продаж, реализовать первыми)
- **P1** — Высокий приоритет (реализовать во втором этапе)
- **P2** — Средний приоритет
- **P3** — Перспективные

# 1. AI Route Guardian — Контроль водителя на маршруте (P0)

**Компоненты:** Backend (AI-сервис), Mobile (push), Frontend (дашборд алармов), Admin (мониторинг)

## Описание

AI-агент, который в реальном времени мониторит каждую активную перевозку и генерирует алармы при отклонениях от нормального поведения. Заказчик и логист перевозчика получают мгновенные уведомления.

## Алармы

### 1.1. Незапланированная остановка
- **Триггер:** Водитель стоит >15 мин вне точки загрузки/разгрузки
- **Данные:** `gis.driver_location` (скорость = 0, координаты не в радиусе точек `applications.cargo_loading`)
- **Серьёзность:** MEDIUM
- **Действие:** Push заказчику + логисту перевозчика + отметка на карте мониторинга

### 1.2. Отклонение от маршрута
- **Триггер:** Водитель ушёл >2 км от оптимального маршрута между точками
- **Данные:** Текущие координаты vs. оптимальный маршрут (Yandex Router API)
- **Серьёзность:** HIGH
- **Действие:** Push + Telegram + email заказчику

### 1.3. Отставание от расчётного времени
- **Триггер:** Прогнозируемое ETA отличается от планового >20%
- **Данные:** Текущая позиция + средняя скорость vs. плановое `loadingDatetime` в `cargo_loading`
- **Серьёзность:** MEDIUM
- **Действие:** Push заказчику с новым ETA

### 1.4. Превышение скорости
- **Триггер:** Скорость выше допустимой для типа груза/дороги
- **Данные:** `gis.driver_location.speed` + тип груза из `applications.transportation`
- **Пороги:**
  - Стандартный груз: >90 км/ч
  - Опасный/хрупкий груз: >70 км/ч
  - Тяжеловесный: >60 км/ч
- **Серьёзность:** HIGH
- **Действие:** Push водителю + логисту + запись в историю

### 1.5. Подозрительное поведение
- **Триггер:** Зигзагообразное движение, частые остановки (>5 за 30 мин), движение в обратном направлении
- **Данные:** Анализ паттерна координат в `gis.driver_location` за скользящее окно
- **Серьёзность:** HIGH
- **Действие:** Telegram + email + пометка перевозки для проверки

### 1.6. Потеря связи
- **Триггер:** Нет GPS-данных >10 мин (при ожидаемом интервале 30 сек)
- **Данные:** Timestamp последней записи в `gis.driver_location`
- **Серьёзность:** CRITICAL
- **Действие:** Push + SMS + Telegram

### 1.7. SOS-паттерн (автоматический)
- **Триггер:** Резкая остановка (скорость >60 → 0 за <5 сек) + нехарактерное место
- **Данные:** Анализ dv/dt (ускорение) по данным speed/timestamp
- **Серьёзность:** CRITICAL
- **Действие:** Автоматическое оповещение всех контактных лиц

## Используемые данные

| Таблица | Поля | Назначение |
|---------|------|------------|
| `gis.driver_location` | location, timestamp, speed, heading, accuracy | GPS-поток в реальном времени |
| `gis.veh_latest_loc` | location, timestamp, status, is_sos | Текущая позиция транспорта |
| `applications.cargo_loading` | location, address, loading_datetime | Точки маршрута (геозоны) |
| `applications.transportation` | status, cargo_name, deadline_date | Контекст перевозки |
| `gis.transportation_route_history` | route_distance | Плановый маршрут |

## Логика работы

```
GPS Update -->
  |-- Проверка геозоны (в точке загрузки/разгрузки?)
  |-- Проверка скорости (допустимая для типа груза?)
  |-- Проверка маршрута (в коридоре оптимального маршрута?)
  |-- Проверка ETA (укладывается в план?)
  |-- Проверка паттерна (зигзаги, обратное направление?)
  +-- Проверка связи (регулярные обновления?)
        |
        +-- При срабатывании --> Alarm Service
              |-- Сохранение аларма в БД
              |-- Push-уведомление (FCM)
              |-- Telegram (канал мониторинга)
              |-- Email (критические алармы)
              +-- Обновление дашборда в реальном времени (WebSocket)
```

## UI-компоненты

### Mobile (заказчик/логист)
- Push-уведомления с типом аларма и ссылкой на карту
- Красная метка на карте в месте аларма
- История алармов в деталях перевозки

### Frontend (веб-кабинет)
- Панель алармов на дашборде (список активных алармов)
- Карта с визуализацией алармов
- Фильтрация по типу/серьёзности
- Звуковое оповещение для критических алармов

### Admin (панель администратора)
- Агрегированная статистика алармов по перевозчикам
- Рейтинг "проблемных" водителей/перевозчиков
- Настройка порогов алармов

## Новые таблицы БД

```sql
CREATE TABLE gis.transportation_alarm (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    transportation_id UUID NOT NULL REFERENCES applications.transportation(id),
    alarm_type VARCHAR(50) NOT NULL, -- UNPLANNED_STOP, ROUTE_DEVIATION, ETA_DELAY, SPEEDING, SUSPICIOUS_BEHAVIOR, CONNECTION_LOST, AUTO_SOS
    severity VARCHAR(20) NOT NULL, -- LOW, MEDIUM, HIGH, CRITICAL
    location GEOGRAPHY(POINT, 4326),
    description TEXT,
    metadata JSONB, -- доп. данные: скорость, отклонение в км, задержка в минутах
    is_resolved BOOLEAN DEFAULT FALSE,
    resolved_at TIMESTAMP,
    resolved_by UUID,
    resolution_note TEXT,
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_alarm_transportation ON gis.transportation_alarm(transportation_id);
CREATE INDEX idx_alarm_type ON gis.transportation_alarm(alarm_type);
CREATE INDEX idx_alarm_created ON gis.transportation_alarm(created_at DESC);
```

## Метрики успеха

- Среднее время реакции на аларм: <5 мин
- % ложных срабатываний: <10%
- Охват: 100% активных перевозок
- Снижение инцидентов на маршруте: -30% за 3 месяца

## Технические требования

- Обработка GPS-обновления: <500 мс
- Задержка доставки аларма: <10 сек
- Хранение алармов: 12 месяцев
- Оптимальный маршрут: кешировать на старте перевозки (Yandex Router API)

# 2. AI ETA Predictor — Прогноз времени доставки (P0)

**Компоненты:** Backend (ML-модель), Mobile (отображение ETA), Frontend (отображение ETA)

## Описание

ML-модель, которая предсказывает точное время прибытия на каждую точку маршрута. Динамически обновляется по мере движения водителя. Заказчик видит не просто "В пути", а "Прибудет через 2ч 15мин (точность 92%)".

## Как работает

### Входные данные (features)

**Статические (на момент создания перевозки):**
- Расстояние между точками маршрута (из `route_distance`)
- Тип груза и вес (`cargo_name`, `cargo_weight`)
- Тип кузова (`vehicle_body_type_id`)
- День недели, время суток
- Сезон (зима/лето — влияет на скорость в Казахстане)
- Регион (горная местность, степь, городские маршруты)
- Водитель — средняя скорость по истории (`gis.driver_location`)
- Перевозчик — исторический % выполнения в срок

**Динамические (обновляются в реальном времени):**
- Текущая позиция водителя
- Средняя скорость за последние 30 мин
- Оставшееся расстояние до следующей точки
- Текущее время (учёт ночного/дневного вождения)
- Кол-во оставшихся точек загрузки/разгрузки
- Время на загрузку/разгрузку (из исторических данных)

### Модель

**Подход:** Gradient Boosted Trees (XGBoost/LightGBM)
- Обучение на исторических данных завершённых перевозок
- Target: фактическое время прибытия на точку
- Регулярное переобучение (еженедельно)

**Baseline:** Расстояние / средняя скорость — простая формула для сравнения.

**Advanced:** Учёт паттернов конкретного водителя, сезонности, типа маршрута.

### Расчёт точности (confidence)

```
confidence = 1 - (MAE / mean_actual_time)

Где:
- MAE = средняя абсолютная ошибка модели на валидации
- Рассчитывается per route_type (город, межгород, дальнобой)
```

- Доступность API: 99.9%
- Улучшение vs. baseline (расстояние/скорость): >30%

## План обучения модели

1. **Сбор данных:** Минимум 500 завершённых перевозок с GPS-треками
2. **Feature engineering:** Извлечение признаков из GPS, маршрутов, грузов
3. **Обучение:** Train/validation/test split 70/15/15
4. **A/B тест:** Показывать ETA для 50% пользователей, сравнить удовлетворённость
5. **Итерация:** Еженедельное переобучение с новыми данными

# 3. Route Intelligence — Аналитика и оптимизация маршрутов (P0)

**Компоненты:** Backend (аналитика), Frontend (дашборд), Admin (отчёты)

## Описание

AI-система аналитики маршрутов, которая анализирует все завершённые перевозки и выдаёт actionable-инсайты: оптимальные маршруты, сезонные паттерны, рекомендации по ценам и загрузке.

## Модули

### 3.1. Оптимальные маршруты по направлениям

**Что делает:** Для каждого популярного направления (Алматы-Астана, Алматы-Шымкент и т.д.) определяет:
- Самый быстрый маршрут (минимальное среднее время)
- Самый дешёвый маршрут (минимальная средняя стоимость за тонну-км)
- Самый надёжный маршрут (минимальный разброс времени)

**Данные:**
- `gis.driver_location` — фактические GPS-треки
- `applications.transportation_cost` — стоимость перевозок
- `applications.cargo_loading` — точки маршрута
- `gis.transportation_route_history` — расстояния

**Алгоритм:**
1. Кластеризация перевозок по направлениям (DBSCAN по начальной и конечной точкам)
2. Для каждого кластера: агрегация метрик (время, стоимость, надёжность)
3. Ранжирование маршрутов внутри кластера
4. Обновление: еженедельно

### 3.2. Тепловая карта загруженности

**Что делает:** Визуализация спроса на перевозки по регионам Казахстана.

**Слои карты:**
- Количество заявок по регионам (отправление)
- Количество заявок по регионам (назначение)
- Средняя цена за тонну-км по регионам
- Доступность свободного транспорта

**Данные:**
- `applications.cargo_loading.location` — геокоординаты точек
- `gis.veh_latest_loc` — позиции свободного транспорта
- `applications.transportation_cost` — цены

**Обновление:** ежедневно

### 3.3. Сезонные паттерны

**Что делает:** Выявляет и прогнозирует сезонные тренды.

**Паттерны:**
- Сельхоз грузы: пик весна (посевная) и осень (уборочная)
- Стройматериалы: пик лето
- Потребительские товары: пик перед праздниками (Наурыз, Новый Год)
- Нефтепродукты: стабильный спрос с зимним пиком

**Алгоритм:**
1. Декомпозиция временных рядов (STL decomposition) по типам грузов
2. Выделение тренда, сезонности, остатка
3. Прогноз на 1-3 месяца вперёд (Prophet / ARIMA)

**Данные:**
- `applications.transportation` — даты, тип груза, маршрут
- `applications.transportation_cost` — стоимость
- Временной ряд: агрегация по неделям/месяцам

### 3.4. Рекомендации по ценообразованию

**Что делает:** Подсказывает оптимальную цену при создании заявки.

**Факторы:**
- Историческая цена по направлению и типу груза
- Текущий спрос/предложение на направлении
- Сезонный коэффициент
- Вес и объём груза
- Срочность (дедлайн)
- Доступность транспорта в регионе отправления

**Отображение:**
```
Рекомендуемая цена: 450 000 тг
Диапазон рынка: 380 000 - 520 000 тг
|-- Дешевле рынка: быстрый подбор, ниже качество
|-- В рынке: оптимальный баланс
+-- Дороже рынка: премиум перевозчики, высокая надёжность
```

### 3.5. Обнаружение "мёртвых пробегов"

**Что делает:** Находит транспорт, который едет порожняком, и рекомендует попутные грузы.

**Алгоритм:**
1. Определить транспорт со статусом FINISHED, едущий в базу
2. Найти открытые заявки по направлению возврата
3. Матчинг по типу кузова, грузоподъёмности, временному окну
4. Уведомление перевозчику: "Попутный груз: {направление}, {цена}"

**Данные:**
- `gis.veh_latest_loc` — позиция и статус транспорта
- `applications.transportation` (status=CREATED) — открытые заявки
- `applications.cargo_loading` — точки загрузки
- `transport.vehicle` — тип кузова, грузоподъёмность

# 4. Smart Matching — Интеллектуальный подбор перевозчика (P1)

**Компоненты:** Backend (ML-модель), Frontend (рекомендации)

## Описание

ML-система, которая автоматически ранжирует перевозчиков для каждой заявки и рекомендует ТОП-3 лучших. Заказчик видит не просто список откликов, а рекомендацию с объяснением.

## Модель скоринга

### Факторы и веса

| Фактор | Вес | Источник данных |
|--------|-----|-----------------|
| Историческая надёжность | 25% | % перевозок в срок из `applications.transportation` |
| KYC-скоринг | 20% | 14 параметров из `users.kyc_score` |
| Близость к точке загрузки | 15% | `gis.veh_latest_loc` vs `cargo_loading.location` |
| Соответствие транспорта | 15% | Тип кузова, грузоподъёмность из `transport.vehicle` |
| Ценовая конкурентоспособность | 10% | Исторические цены из `transportation_cost` |
| Отзывы заказчиков | 10% | `CustomerSatisfaction` score |
| Опыт на направлении | 5% | Кол-во перевозок по аналогичному маршруту |

### Алгоритм

**Этап 1 — Фильтрация (жёсткие критерии):**
- Тип кузова совместим с грузом
- Грузоподъёмность >= вес груза
- Перевозчик активен (status = ACTIVE)
- KYC-скоринг >= минимальный порог

**Этап 2 — Скоринг (мягкие критерии):**
- Каждый фактор нормализуется [0, 1]
- Взвешенная сумма = итоговый match_score
- Ранжирование по match_score DESC

**Этап 3 — Объяснение:**
- Для каждого ТОП-кандидата формируется текст:
  - "98% перевозок в срок" (надёжность)
  - "Свободный транспорт в 45 км от загрузки" (близость)
  - "Средняя цена на 8% ниже рынка" (цена)

### ML-модель (advanced)

После накопления данных об откликах и выборах заказчиков:

**Подход:** Learning-to-Rank (LambdaMART)
- **Features:** Все факторы скоринга + взаимодействия
- **Target:** Был ли перевозчик выбран заказчиком (binary)
- **Обучение:** На исторических данных "отклик -> выбор"
- **Переобучение:** ежемесячно

## Источники данных

| Таблица | Использование |
|---------|---------------|
| `users.kyc_score` | 14-мерный скоринг перевозчика |
| `users.company_check_data` | Проверка компании (налоги, суды, кредит) |
| `users.person_check_data` | Проверка водителя (ДТП, штрафы, лицензия) |

# 5. Demand Forecasting — Прогноз спроса на перевозки (P1)

**Компоненты:** Backend (ML-модель), Frontend (дашборд), Mobile (уведомления)

## Описание

ML-система прогнозирования спроса на перевозки по направлениям, типам грузов и временным периодам. Помогает перевозчикам планировать загрузку, а заказчикам — выбирать оптимальное время.

## Модули

### 5.1. Прогноз объёма заявок

**Гранулярность:**
- По направлениям (пары регионов)
- По типам грузов
- По неделям (горизонт 4 недели)

**Алгоритм:**
- **Baseline:** Prophet (Facebook) — хорошо работает с сезонностью и праздниками
- **Advanced:** LightGBM с lag-features + календарные фичи

**Features:**
- Исторический объём заявок (lag 1-52 недели)
- День недели, неделя года
- Праздники Казахстана (Наурыз, Новый Год, День Конституции и др.)
- Сезон (посевная, уборочная, строительный сезон)
- Тип груза
- Регион отправления/назначения
- Средняя цена за период (прокси спроса)

**Данные:**
- `applications.transportation` — дата создания, тип груза
- `applications.cargo_loading` — регион загрузки/разгрузки
- `applications.transportation_cost` — стоимость

### 5.2. Прогноз цен

**Что показывает:** Ожидаемый диапазон цен на перевозки по направлениям на ближайшие недели.

**Алгоритм:** Quantile Regression (предсказание 10-го, 50-го, 90-го перцентилей цены)

**Features:**
- Прогнозируемый объём заявок (из модуля 1)
- Исторические цены по направлению
- Сезонность
- Доступность транспорта в регионе

### 5.3. Рекомендации для перевозчиков

На основе прогнозов формируются персонализированные рекомендации:

```
"На следующей неделе ожидается рост заявок Алматы->Астана на 30%.
Рекомендуем перенаправить 2 машины в Алматинскую область.
Ожидаемая цена: 420-480 тыс. тг (рост 12% к текущей неделе)"
```

### 5.4. Рекомендации для заказчиков

```
"Сейчас высокий сезон на направлении Шымкент->Алматы.
Если можете подождать 2 недели, цена снизится на ~15%.
Рекомендуемая цена сейчас: 380 тыс. тг"
```

# 6. Credit AI — Скоринг для факторинга (P1)

**Компоненты:** Backend (ML-модель), Frontend (отображение скора), Admin (управление)

## Описание

AI-система оценки кредитоспособности перевозчиков для факторинговых операций. Автоматически рассчитывает риск неплатежа и рекомендуемый лимит финансирования.

## Текущее состояние

Платформа уже имеет богатую базу для скоринга:
- **14-мерный KYC-скоринг** (`users.kyc_score`) — налоги, суды, кредит, ДТП, штрафы и др.
- **Проверка компании** (`users.company_check_data`) — налоговая задолженность, судебные дела, кредитная история, лицензии
- **Проверка водителя** (`users.person_check_data`) — лишение прав, ДТП, судимость, медицинские ограничения
- **История платежей** — данные по оплатам и задержкам
- **Факторинговые операции** (`factoring.*`) — история пользования факторингом

**Проблема:** Решение о факторинге принимается вручную. AI может автоматизировать и улучшить точность.

## Модель

### Входные данные (features)

**KYC-блок (из `users.kyc_score`):**
- legal_status_score — юридический статус
- tax_discipline_score — налоговая дисциплина
- lawsuits_score — судебные дела
- credit_history_score — кредитная история
- factoring_score — история факторинга
- dtp_fines_score — ДТП и штрафы
- customer_reviews_score — отзывы заказчиков
- order_history_score — история заказов
- documents_score — полнота документов

**Финансовый блок:**
- Средний объём перевозок за 3/6/12 месяцев
- Средняя стоимость перевозки
- Средняя задержка оплаты (payment_delay из `transportation_cost`)
- Процент перевозок с задержкой оплаты
- Текущая дебиторская задолженность
- История использования факторинга (суммы, возвраты)

**Операционный блок:**
- Количество завершённых перевозок
- % перевозок в срок
- % отменённых перевозок
- Количество SOS-инцидентов
- Давность регистрации на платформе

### Целевая переменная

- **Binary:** Был ли дефолт (задержка >60 дней) по факторинговой операции
- **Regression:** Количество дней задержки оплаты

### Алгоритм

**Модель:** Gradient Boosting (XGBoost)
- Устойчив к пропускам в данных
- Хорошо работает с табличными данными
- Интерпретируемость через SHAP values

**Выход:**
```
credit_score: 0-100  (0 = высокий риск, 100 = низкий риск)
risk_category: LOW / MEDIUM / HIGH / CRITICAL
recommended_limit: сумма в тенге
confidence: 0.0-1.0
```

### Расчёт лимита

```
base_limit = avg_monthly_revenue * 0.7
risk_multiplier = credit_score / 100
recommended_limit = base_limit * risk_multiplier
max_limit = cap по политике компании
final_limit = min(recommended_limit, max_limit)
```

# 7. Дополнительные AI-фичи

## P2 — Средний приоритет

### 7.1. Мониторинг усталости водителя (Driver Fatigue Detection)

**Описание:** AI-система отслеживает режим труда и отдыха водителя по GPS-данным.

**Алармы:**
- Непрерывное вождение >4 часов без остановки >15 мин
- Суммарное вождение >10 часов за сутки
- Ночное вождение (23:00-06:00) >4 часов
- Менее 8 часов между сменами

**Данные:**
- `gis.driver_location` — timestamp + speed (определение периодов движения/отдыха)
- `gis.employee_latest_loc` — always-on tracking между заказами

**Нормативная база:** Правила режима труда и отдыха водителей РК (Приказ МТК РК).

**Реализация:**
- Анализ скользящего окна GPS-данных
- Определение периодов движения (speed > 5 км/ч) и отдыха
- Push водителю: "Рекомендуем остановиться на отдых"
- Аларм логисту при нарушении нормы

---

### 7.2. Автоматическая проверка документов (Document AI)

**Описание:** OCR + AI для автоматической верификации загружаемых документов.

**Документы:**
- Удостоверение личности -> извлечение ФИО, ИИН, срок действия
- Водительское удостоверение -> категории, срок действия
- Тех. паспорт -> марка, гос. номер, VIN
- Доверенность -> сроки, полномочия

**Технологии:**
- Google Cloud Vision / Yandex Vision для OCR
- Кастомная модель классификации типа документа
- Rule-based валидация извлечённых полей

**Результат:**
- Автозаполнение полей профиля при загрузке документа
- Валидация: данные в документе совпадают с данными в профиле
- Алерт при истечении срока документа за 30 дней

**Данные:**
- `file.file_meta_info` — загруженные файлы
- `users.employee` — данные профиля для сверки
- `transport.vehicle` — данные транспорта для сверки

---

### 7.3. Умные уведомления (Smart Notifications)

**Описание:** AI-оптимизация доставки уведомлений.

**Оптимальное время отправки:**
- Анализ `notifications.notification_log` (opened, clicked по времени)
- Модель предсказания вероятности прочтения в зависимости от времени
- Отправка в персональное "окно внимания" каждого пользователя

**Оптимальный канал:**
- Для каждого пользователя определить предпочтительный канал
- Push vs SMS vs Email — по conversion rate из исторических данных
- Каскад: сначала Push, если не прочитал за 15 мин -> SMS

**Персонализация контента:**
- A/B тестирование шаблонов уведомлений
- Подбор tone of voice (формальный/неформальный) по engagement

**Данные:**
- `notifications.notification_log` — channel, status, opened, clicked, delivery_attempts
- `notifications.user_delivery_channels` — предпочтения + quiet hours

### 7.4. Детектор аномалий и мошенничества (Fraud Detection)

**Описание:** AI-система обнаружения подозрительных паттернов.

**Накрутка пробега:**
- Фактический GPS-трек значительно длиннее оптимального маршрута
- Алгоритм: сравнение `route_distance` из GPS vs. Yandex Router
- Порог: отклонение >30%

**Фиктивные перевозки:**
- GPS-трек не соответствует заявленному маршруту
- Слишком быстрое "завершение" перевозки
- Одинаковые GPS-паттерны у разных перевозок одного перевозчика

**Ценовой сговор:**
- Аномально высокие цены от группы перевозчиков на одном направлении
- Статистический анализ распределения цен + обнаружение кластеров

**Двойные SOS:**
- Частые SOS-сигналы от одного водителя/перевозчика
- Анализ паттерна: SOS -> отмена -> новая перевозка с наценкой

**Реализация:**
- Isolation Forest для обнаружения аномалий в GPS-треках
- Statistical process control для ценовых аномалий
- Rule-based + ML ensemble для комплексной оценки

**Данные:**
- `gis.driver_location` — GPS-треки
- `applications.transportation_cost` — цены
- `applications.transportation` — статусы, сроки

### 8.3. Route Guardian (real-time processing)

**Особенность:** Работает в реальном времени, обрабатывает каждое GPS-обновление.

**v1 (простой):** Обработка в Spring Boot.
- `DriverLocationService` при каждом GPS update вызывает `RouteGuardianService`
- Проверки выполняются синхронно
- Алармы сохраняются в БД и отправляются через `NotificationService`

**v2 (масштабируемый):** Event-driven.
- GPS updates публикуются в Kafka/Redis Stream
- AI Microservice подписывается на стрим
- Алармы публикуются обратно
- Spring Boot читает алармы и уведомляет пользователей

**Рекомендация:** Начать с v1, перейти на v2 при >1000 одновременных перевозок.

### 8.4. Data Layer

**PostgreSQL (основная БД):**
- Все существующие данные
- Новые таблицы: `gis.transportation_alarm`, `ai.prediction_log`, `ai.model_metrics`

**Redis:**
- Кеш ETA предсказаний (TTL 5 мин)
- Кеш matching скоров (TTL 1 час)
- Кеш кредитных скоров (TTL 24 часа)
- Real-time данные для Route Guardian (последние N точек водителя)
- Оптимальные маршруты (кеш Yandex Router)

**TimescaleDB (опционально, v2):**
- Для больших объёмов GPS-данных (>10 млн записей/месяц)
- Автоматическое сжатие старых данных
- Быстрые time-series запросы

## Мониторинг AI-системы

### Метрики для отслеживания
- **Latency:** p50, p95, p99 времени ответа каждого AI-endpoint
- **Accuracy:** MAPE для ETA, AUC-ROC для credit scoring, precision/recall для fraud
- **Drift:** Изменение распределения входных данных vs. обучающих
- **Usage:** Кол-во вызовов, adoption rate рекомендаций

### Дашборд мониторинга (Admin)
- Статус моделей (версия, дата обучения, метрики)
- Графики accuracy за время
- Алерты при деградации качества
- Feature importance для каждой модели

## Требования к инфраструктуре

### Минимальные (MVP)
- Дополнительная нагрузка на Spring Boot: +10-15% CPU
- Redis: +512 MB RAM
- PostgreSQL: +2-5 GB для алармов и логов

### Для Python Microservice (Этап 2+)
- 1 instance: 2 CPU, 4 GB RAM
- GPU не требуется (табличные модели, не deep learning)
- Хранение моделей: ~100 MB
- Docker deployment (рядом с основным бэкендом)

### Для масштабирования (Этап 4+)
- 2-3 instances Python Microservice (за балансером)
- Kafka/Redis Streams для event-driven обработки
- TimescaleDB для time-series данных
- MLflow для model registry
