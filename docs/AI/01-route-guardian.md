# AI Route Guardian — Контроль водителя на маршруте

**Приоритет:** P0
**Компоненты:** Backend (AI-сервис), Mobile (push), Frontend (дашборд алармов), Admin (мониторинг)

## Описание

AI-агент, который в реальном времени мониторит каждую активную перевозку и генерирует алармы при отклонениях от нормального поведения. Заказчик и логист перевозчика получают мгновенные уведомления.

## Алармы

### 1. Незапланированная остановка
- **Триггер:** Водитель стоит >15 мин вне точки загрузки/разгрузки
- **Данные:** `gis.driver_location` (скорость = 0, координаты не в радиусе точек `applications.cargo_loading`)
- **Серьёзность:** MEDIUM
- **Действие:** Push заказчику + логисту перевозчика + отметка на карте мониторинга

### 2. Отклонение от маршрута
- **Триггер:** Водитель ушёл >2 км от оптимального маршрута между точками
- **Данные:** Текущие координаты vs. оптимальный маршрут (Yandex Router API)
- **Серьёзность:** HIGH
- **Действие:** Push + Telegram + email заказчику

### 3. Отставание от расчётного времени
- **Триггер:** Прогнозируемое ETA отличается от планового >20%
- **Данные:** Текущая позиция + средняя скорость vs. плановое `loadingDatetime` в `cargo_loading`
- **Серьёзность:** MEDIUM
- **Действие:** Push заказчику с новым ETA

### 4. Превышение скорости
- **Триггер:** Скорость выше допустимой для типа груза/дороги
- **Данные:** `gis.driver_location.speed` + тип груза из `applications.transportation`
- **Пороги:**
  - Стандартный груз: >90 км/ч
  - Опасный/хрупкий груз: >70 км/ч
  - Тяжеловесный: >60 км/ч
- **Серьёзность:** HIGH
- **Действие:** Push водителю + логисту + запись в историю

### 5. Подозрительное поведение
- **Триггер:** Зигзагообразное движение, частые остановки (>5 за 30 мин), движение в обратном направлении
- **Данные:** Анализ паттерна координат в `gis.driver_location` за скользящее окно
- **Серьёзность:** HIGH
- **Действие:** Telegram + email + пометка перевозки для проверки

### 6. Потеря связи
- **Триггер:** Нет GPS-данных >10 мин (при ожидаемом интервале 30 сек)
- **Данные:** Timestamp последней записи в `gis.driver_location`
- **Серьёзность:** CRITICAL
- **Действие:** Push + SMS + Telegram

### 7. SOS-паттерн (автоматический)
- **Триггер:** Резкая остановка (скорость >60 → 0 за <5 сек) + нехарактерное место
- **Данные:** Анализ dv/dt (ускорение) по данным speed/timestamp
- **Серьёзность:** CRITICAL
- **Действие:** Автоматическое оповещение всех контактных лиц

## Используемые данные

| Таблица | Поля | Назначение |
|---------|------|------------|
| `gis.driver_location` | location, timestamp, speed, heading, accuracy | GPS-поток в реальном времени |
| `gis.veh_latest_loc` | location, timestamp, status, is_sos | Текущая позиция транспорта |
| `applications.cargo_loading` | location, address, loading_datetime | Точки маршрута (геозоны) |
| `applications.transportation` | status, cargo_name, deadline_date | Контекст перевозки |
| `gis.transportation_route_history` | route_distance | Плановый маршрут |

## Логика работы

```
GPS Update →
  ├─ Проверка геозоны (в точке загрузки/разгрузки?)
  ├─ Проверка скорости (допустимая для типа груза?)
  ├─ Проверка маршрута (в коридоре оптимального маршрута?)
  ├─ Проверка ETA (укладывается в план?)
  ├─ Проверка паттерна (зигзаги, обратное направление?)
  └─ Проверка связи (регулярные обновления?)
      │
      └─ При срабатывании → Alarm Service
            ├─ Сохранение аларма в БД
            ├─ Push-уведомление (FCM)
            ├─ Telegram (канал мониторинга)
            ├─ Email (критические алармы)
            └─ Обновление дашборда в реальном времени (WebSocket)
```

## UI-компоненты

### Mobile (заказчик/логист)
- Push-уведомления с типом аларма и ссылкой на карту
- Красная метка на карте в месте аларма
- История алармов в деталях перевозки

### Frontend (веб-кабинет)
- Панель алармов на дашборде (список активных алармов)
- Карта с визуализацией алармов
- Фильтрация по типу/серьёзности
- Звуковое оповещение для критических алармов

### Admin (панель администратора)
- Агрегированная статистика алармов по перевозчикам
- Рейтинг "проблемных" водителей/перевозчиков
- Настройка порогов алармов

## Новые таблицы БД

```sql
CREATE TABLE gis.transportation_alarm (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    transportation_id UUID NOT NULL REFERENCES applications.transportation(id),
    alarm_type VARCHAR(50) NOT NULL, -- UNPLANNED_STOP, ROUTE_DEVIATION, ETA_DELAY, SPEEDING, SUSPICIOUS_BEHAVIOR, CONNECTION_LOST, AUTO_SOS
    severity VARCHAR(20) NOT NULL, -- LOW, MEDIUM, HIGH, CRITICAL
    location GEOGRAPHY(POINT, 4326),
    description TEXT,
    metadata JSONB, -- доп. данные: скорость, отклонение в км, задержка в минутах
    is_resolved BOOLEAN DEFAULT FALSE,
    resolved_at TIMESTAMP,
    resolved_by UUID,
    resolution_note TEXT,
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_alarm_transportation ON gis.transportation_alarm(transportation_id);
CREATE INDEX idx_alarm_type ON gis.transportation_alarm(alarm_type);
CREATE INDEX idx_alarm_created ON gis.transportation_alarm(created_at DESC);
```

## Метрики успеха

- Среднее время реакции на аларм: <5 мин
- % ложных срабатываний: <10%
- Охват: 100% активных перевозок
- Снижение инцидентов на маршруте: -30% за 3 месяца

## Технические требования

- Обработка GPS-обновления: <500 мс
- Задержка доставки аларма: <10 сек
- Хранение алармов: 12 месяцев
- Оптимальный маршрут: кешировать на старте перевозки (Yandex Router API)
